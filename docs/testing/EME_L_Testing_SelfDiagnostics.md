# Самодиагностика и системные тесты

Синонимы: самодиагностика, RobotHealer, healClasses, KPI, тест здоровья БД, DBWatch, дозор базы данных, CIRobot, роботы-тестировщики, UI-тест, is_run_dbtest, is_vfs_test, is_test_block_write, is_ui_test_mode, is_test_references_counter, is_trace_query, is_lost_fields

## Самодиагностика (RobotHealer)

Система самодиагностики (`RobotHealer.healClasses.*`) — это механизм периодической проверки здоровья базы данных и ключевых операций WMS. Каждый тест оценивает состояние по шкале от 1 до 10 и при необходимости выполняет исправляющие действия.

Это **не классические тесты** в духе `assertEquals`, а **KPI-мониторинг**, который:
- Проверяет скорость операций
- Выявляет некорректные данные в БД
- Автоматически исправляет найденные проблемы
- Отправляет отчёты по почте

### Контракт healClasses

Каждый класс самодиагностики должен реализовывать три метода:

| Метод | Назначение | Возвращает |
|-------|------------|------------|
| `describe()` | Название тестируемой функции (для пользователя) | `String` |
| `test()` | Выполняет тест и ставит оценку | `Integer` (1–10) |
| `heal()` | Выполняет исправление проблем | `String` (отчёт) |

### Шаблон: `testExample`

Файл: `testExample.txt`

```eme-l
RobotHealer.healClasses.testExample
************************             Флаги             ************************
ДЛ: Нет
НК: Нет
ВЗ: Нет
************************          Конструктор          ************************

************************             Методы            ************************

/* #KPI 13.09.2013 Пример класса для тестирования производительности одной функции программы */

/* метод возвращает название тестируемой функции */
describe()
{
    return tr("Вписать здесь название тестируемой функции");
}

/* метод выполняет тест и возвращает оценку */
test()
{
    rating = 1;
    ' TO DO тест должен поставить оценку от одного до десяти '
    rating = rating + SameTest1();
    rating = rating + SameTest2();
    Return rating;
}

/* метод выполняет необходимые действия для исправления проблем */
heal()
{
    strResult = "";

    is_transaction(1); ' тут нужно дать транзакции нормальное имя '
    ' TO DO вызов исправляющей функции '
    strResult = strResult + CureSameTest1();
    strResult = strResult + CureSameTest2();
    is_transaction(-1);

    Return strResult;
}
```

### Реальные тесты самодиагностики

#### `testDebug` — проверка режима отладки

```eme-l
testDebug
{
    describe()
    {
        return tr("Проверка режима отладки");
    }

    test()
    {
        If (d_emel)
            Return 10;  ' Отладка включена — критично для продакшена'
        Else
            Return 1;   ' Отладка выключена — норма'
        End If
    }

    heal()
    {
        d_emel = FALSE;
        Return tr("Режим отладки отключен");
    }
}
```

#### `testChainRestore` — восстановление цепочек документов

Проверяет разрывы цепочек в `DocLines` (ссылки `Prev`/`Next`). Восстанавливает связи по дате отгрузки.

#### `testAccessibility` — доступность ячеек

```eme-l
testAccessibility
{
    describe()
    {
        return tr("Проверка доступности ячеек");
    }

    test()
    {
        ' Проверяет CorrectAccessibility и скорость проверки '
        ' Возвращает оценку 1–10 в зависимости от состояния '
    }

    heal()
    {
        ' Исправляет флаги доступности ячеек '
    }
}
```

Проверяет корректность флагов доступности ячеек (`CorrectAccessibility`) и скорость проверки.

#### `testFindCell` — поиск ячеек

Проверяет скорость автопоиска ячеек для размещения:
- `MaximumMURef` — вместимость по объёму
- Вместимость по высоте
- Фильтры стратегий размещения

#### `testFindNonAddrCell` — безадресные ячейки

Проверяет состояние безадресных ячеек:
- Пустые регистры
- Неправильный тип ячейки
- Переполненность

#### `testRegisters` — фрагментация регистров

Проверяет фрагментацию записей регистров (блоки по 1024 строки). Содержит процедуру дефрагментации.

#### `testKernelSettings` — параметры ядра

«Кровавая инструкция» по настройкам сервера:
- Архивы
- HTTP-потоки
- DLL
- Транзакции
- Кэш

#### `testStructure` — корректность структуры БД

Проверяет корректность сортировочных полей-цепочек в структуре БД (ссылка на заголовок `999`).

#### `testSelfTest` — работоспособность модуля

Проверяет работоспособность модуля самодиагностики, включая отправку почты через SMTP.

## Дозор базы данных (DBWatch)

`DBWatch` — механизм отслеживания изменений в БД между подготовкой данных и записью в транзакции. Используется для предотвращения конфликтов при параллельном доступе.

### Пример с блокировкой

```eme-l
Tests.TestDBWatch
************************             Флаги             ************************
ДЛ: Нет
НК: Да
ВЗ: Нет
************************          Конструктор          ************************
m_DBWatch = Object("DBWatch");
m_TrueSign;
m_Line;

************************             Методы            ************************
/* Подготовка данных вне транзакции */
OnPrepare()
{
    r_GoodsItem = Object("dsDB", "GoodsItem");
    r_GoodsItem.SetFirstLine();
    m_TrueSign = FALSE;
    If (r_GoodsItem.IsValidLine() & ~r_GoodsItem.IsTrueSign())
        m_TrueSign = TRUE;
        m_Line = r_GoodsItem.GetLine();
    End If
    'Добавляем поле GlobalFlagsThreeFld на текущей строке в наблюдение'
    m_DBWatch.WatchFieldLine(r_GoodsItem.GetGlobalFlagsThreeFld());
}

/* Запись внутри транзакции */
OnChange()
{
    If (m_TrueSign)
        r_GoodsItem = Object("dsDB", "GoodsItem");
        r_GoodsItem.SetLine(m_Line);
        r_GoodsItem.SetTrueSign();
    End If
}

/* Полный цикл с блокировкой */
Do()
{
    'Начинаем дозор (до чтения данных БД)'
    'is_db_watch поднимает данные на верхний уровень'
    is_db_watch(1);

    'Проводим подготовку данных вне транзакции'
    If (is_workstation_mode() != "MONO")
        OnPrepare();
    End If

    'Открываем транзакцию'
    is_transaction(1, "Устанавливаем ЧЗ");

    'Завершаем дозор — больше информация не собирается'
    is_db_watch(-1);

    'Проверяем изменения: если есть — делаем подготовку заново'
    If (is_workstation_mode() == "MONO" | m_DBWatch.Check() != "")
        OnPrepare();
    End If

    'Производим изменения в БД на основе ранее подготовленных данных'
    OnChange();

    'Закрываем транзакцию'
    is_transaction(-1);
}

/* Простой вариант без блокировки */
Do2()
{
    is_transaction(1, "Устанавливаем ЧЗ");
    r_GoodsItem = Object("dsDB", "GoodsItem");
    r_GoodsItem.SetFirstLine();
    If (~r_GoodsItem.IsTrueSign())
        r_GoodsItem.SetTrueSign();
    End If
    is_transaction(-1);
}
```

**Паттерн DBWatch:**
1. `is_db_watch(1)` — начать наблюдение (до чтения данных)
2. `OnPrepare()` — читаем данные, запоминаем состояние
3. `m_DBWatch.WatchFieldLine(...)` — добавляем поля в наблюдение
4. `is_transaction(1)` — открываем транзакцию
5. `is_db_watch(-1)` — завершаем наблюдение
6. `m_DBWatch.Check()` — проверяем, изменились ли данные
7. Если изменились — повторяем `OnPrepare()`
8. `OnChange()` — записываем изменения
9. `is_transaction(-1)` — закрываем транзакцию

## UI-тест интерфейса диалогов

**Запуск:** `Система → Ресурсы → Тестирование → Тест интерфейса`

Последовательно открывает все диалоги системы (~3 минуты). Не нажимает кнопки и не закрывает окна сообщений — тест должен пройти без ручного вмешательства.

### Ключевая is-функция: `is_ui_test_mode()`

```eme-l
Dialog_OnLoad()
{
    If (ContextFindParamsRec.GetNoOfLines() == 0 & ~is_ui_test_mode())
        If (is_message(tr("Внимание"), tr("Создать настройки поиска?"), "YESNO", "QUESTION") == "YES")
            CreateDefault();
        End If
    End If
}
```

- `is_ui_test_mode()` — возвращает `TRUE`, если система в режиме тестирования интерфейса
- Используется для скрытия `MessageBox` и браузеров, чтобы не блокировать автотест
- Без этой проверки диалоговые окна остановят UI-тест, ожидая ручного нажатия кнопки

### Проверка потерянных полей: `is_lost_fields`

UI-тест также выполняет проверку полей, использующих `CEMERec`, но отсутствующих в БД (несоответствие метапроекта и данных).

```eme-l
' Проверить совместимость метапроекта с данными
Missing = Object("Array")
TypeErrors = Object("Array")
Dups = Object("Array")
is_lost_fields(Missing, TypeErrors, Dups)

' Если Missing не пуст — есть отсутствующие поля
If (Missing.GetSize() > 0)
    is_error(1, "Обнаружены несоответствия метапроекта")
End If
```

| Параметр | Назначение |
|----------|------------|
| `Missing` (ByRef) | Массив отсутствующих полей. Формат: `имя_класса\|имя_записи\|имя_метода\|имя_поля` |
| `TypeErrors` (ByRef) | Массив ошибок типов. Формат: `имя_класса\|имя_записи\|тип_метода\|тип_поля` |
| `Dups` (ByRef) | Массив дублирующихся методов. Формат: `имя_класса\|имя_метода` |

Тест считается пройденным, если после выполнения не осталось открытых диалогов и сообщений об ошибках.

## Роботы-тестировщики (CIRobot)

**Запуск:** `Система → Ресурсы → Тестирование → Робот`  
**Требуется монопольный режим.**

Роботы автоматически выполняют полные сценарии работы с классификаторами и операциями.

### Типы роботов

| Тип | Что тестирует | Базовый класс |
|-----|---------------|---------------|
| **Классификаторы** | Склады, сектора, товары, транзакции | `GuideRobots` |
| **Операции** | Приход, отгрузка, перемещение, изменение характеристик, списание | `OperationRobots` |
| **Отчёты** | Печатные формы | `ReportsRobots` |

### Запуск робота через код

```eme-l
' Запустить робота для тестирования диалогов
is_run_robot("RobotOpenDialogs", 1000, 600, 0)
```

| Параметр | Описание |
|----------|----------|
| `args[0]` | Имя робота: `"RobotOpenDialogs"`, `"RobotStaffPosition"`, `"RobotStaff"` |
| `args[1]` | Пошаговая задержка в мс (по умолчанию 500) |
| `args[2]` | Таймаут в секундах (по умолчанию 500) |
| `args[3]` | Признак скрытия диалога (по умолчанию 0) |

### Добавление нового робота

1. Создать класс
2. Реализовать метод `RobotBatchGo_RST(Robot AS "CIRobot")`
3. Прописать в конструкторе класса `AllRobots`

Роботы могут вызывать друг друга, передавая параметры через `CRobotDataContainer`.

## Тесты целостности базы данных

### `is_run_dbtest` — полный тест БД

```eme-l
' Полный тест (физический + логический)
Result = is_run_dbtest()

' Только физический тест для записи 10
Result = is_run_dbtest(1, 10)
```

| Параметр | Описание |
|----------|----------|
| `args[0]` | `0` — физический + логический, `1` — только физический, `2` — только логический |
| `args[1]` | Номер записи (по умолчанию все) |
| `args[2]` | Номер поля (по умолчанию все) |

**Возвращает:** `Integer` — `0` = ошибок не обнаружено.

### `is_test_references_counter` — счётчики ссылок

```eme-l
' Проверить счётчики ссылок для всех записей
is_test_references_counter()

' Проверить диапазон записей 10-50
is_test_references_counter(10, 50)
```

Проверяет целостность счётчиков ссылок между записями — ключевой тест для выявления «висячих» ссылок.

## Отладочные is-функции для тестирования

| Функция | Назначение |
|---------|------------|
| `is_memory_leak(1024)` | Создать утечку памяти указанного размера для тестирования |
| `is_OLEAUT32_error()` | Генерация принудительного исключения OLEAUT32 |
| `is_test_email_message(...)` | Отправка email с выбором режима (тест почты) |
| `is_lost_fields(M, T, D)` | Проверка совместимости метапроекта с данными |

### Пример: `is_test_email_message`

```eme-l
' Синхронная отправка для проверки соединения
result = is_test_email_message(
    "test@local",           ' отправитель
    "",                     ' отображаемое имя отправителя
    "admin@local",          ' получатель
    "",                     ' отображаемое имя получателя
    "Тест",                 ' тема
    "Проверка связи",       ' текст
    FALSE,                  ' FALSE = plain text, TRUE = HTML
    "",                     ' путь к вложению
    FALSE                   ' FALSE = синхронно, TRUE = асинхронно
)
' result = 1 при успешной отправке
```

## Требования к юнит-тестам в коде

Согласно стандарту оформления кода (`EME_L_Examples.md`):

> **Должны быть предусмотрены средства автотестирования (юнит-тесты).**

Код должен включать самодиагностику: проверки промежуточных результатов и значений из БД.
