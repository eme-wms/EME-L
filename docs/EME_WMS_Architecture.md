# Архитектура системы EME.WMS и компоненты EME БД

**Описание:** Система EME.WMS — складская информационная система для автоматизации складских процессов. Состоит из сервера БД, рабочих станций, HTTP-сервера, ТСД и Монитора Заданий.
**Сленг:** WMS, складская система, еме-дб, база EME, EME.WMS

## Компоненты системы EME.WMS

Система EME.WMS состоит из нескольких компонентов, каждый из которых выполняет свою роль в обеспечении работы склада.

**Синонимы для поиска:** компоненты EME.WMS, состав системы, сервер WMS, рабочая станция, HTTP-сервер, ТСД, терминал, архитектура WMS

### Основные компоненты EME.WMS

| Компонент | Описание | Сленг |
|-----------|----------|-------|
| БД (База данных) | Структурированное хранилище данных | база, БД |
| Сервер приложений | Обеспечивает бизнес-процессы и целостность данных | сервер, сервер WMS |
| Рабочая станция | Интерфейс пользователя для работы с БД | рабстанция, WS |
| HTTP-сервер | Сервер для клиентских устройств (ТСД, Монитор Заданий) | http-сервер |
| Мозговой компьютер | Выполнение регламентных заданий | мозговой ПК, bc |
| ТСД (Терминал сбора данных) | Мобильное устройство для складских операций | терминал, ТСД |
| Монитор Заданий | Интерфейс для операторов БД | монитор, ТМ |

## Метапроект EME БД

Метапроект представляет собой расширение программ на языке EME-L и C++, обеспечивающее работу одной версии с несколькими проектами.

**Синонимы для поиска:** метапроект, сборка метапроекта, DLL метапроекта, компоненты метапроекта, сборка WMS

### Состав метапроекта

| Компонент | Описание |
|-----------|----------|
| MetC_Sys | Системные классы, интерпретатор EME-L, сервер заданий |
| MetC_WMS | Классы записей БД, бизнес-логика WMS |
| MetD_Sys | Функционал работы с диалогами |
| MetR_Sys | Формирование отчётов и печатных форм |

### Сборка метапроекта

Для сборки метапроекта в Windows используется Visual Studio. Процесс сборки включает:

1. Выгрузка исходников из SVN
2. Настройка переменных среды для Qt и Boost
3. Открытие файла `outerWMS.sln` в Visual Studio
4. Сборка в конфигурации Release x64

```eme-l
'Проверка успешности сборки: проверка наличия DLL'
'После сборки должны появиться:'
'- MetC_Sys.dll'
'- MetC_WMS.dll'
'- MetD_Sys.dll'
'- MetR_Sys.dll'
'- MetC_CLR.dll (опционально, для 3PL)'
```

## Режимы запуска EME БД

EME БД может работать в различных режимах, которые определяют её функционал.

**Синонимы для поиска:** режимы запуска, ключи запуска, параметры запуска EME, mono server ws, конфигурация запуска

### Основные режимы запуска

| Режим | Описание | Ключ |
|-------|----------|------|
| Сервер | Режим сервера транзакций | `/server` |
| Рабочая станция | Режим рабочей станции | `/ws` |
| Моно | Однопользовательский режим | `/mono` |

### Дополнительные ключи запуска

| Ключ | Описание | Сленг |
|------|----------|-------|
| `/http` | Запуск HTTP-сервера | http-режим |
| `/bc` | Запуск мозгового компьютера | брейн-компьютер |
| `/fullvm` | Чтение данных только из локального хранилища | fullvm |
| `/netvm` | Взаимодействие с сервером | netvm |
| `/debug` | Запись в логи отладочной информации, поиск органов | debug |
| `/d_emel` | Отладка EME-L кода | отладка emel |
| `/d_dataslice` | Возможность загружать срезы | срезы, dataslice |
| `/nodllalign` | Без выравнивания по модулям | nodllalign |
| `/d_replica` | Функционал репликации в SQL | репликация |
| `/si` | Отключение будильников | si |
| `/rundbtest` | Полный тест целостности при запуске | dbtest |
| `/d_skipdbtst` | Отключение теста целостности | skip test |
| `/skiphwtest` | Отключение проверки оборудования | skip hw |
| `/dblevel=[N]` | Запуск с данными на заданном номере транзакции | dblevel |
| `/d_random` | Защита целостности при стресс-тестах интерфейса ERP-WMS | random |

Можно указывать неограниченное количество ключей, но некоторые могут конфликтовать (например: `/fullvm` и `/netvm`).

### Особенности моно-режима

- Моно-режим возможен только на полной копии БД (невозможен на копии, запущенной ранее в /netvm)
- Если экземпляр, работавший в /ws, запустить в моно-режиме — все локальные изменения попадут только в локальную БД и на сервер никогда не попадут
- Если экземпляр, работавший в моно-режиме, запустить в /ws — локальная копия БД заменится на копию сервера (произойдёт выравнивание)

### Пример файла eme.config

```ini
' Файл конфигурации запуска EME БД
ShowDialog = 1
EditData = 1

' Конфигурация моно-режима с HTTP и BC
"mono_http_bc" = /mono "C:\EME.WMS\workstation\dat_workstation\eme.hdr" /bc /http /d_dataslice

' Конфигурация сервера
"server_mode" = /server "C:\EME.WMS\server\dat_server\eme.hdr"

' Конфигурация рабочей станции
"ws_mode" = /ws "C:\EME.WMS\workstation\dat_workstation\eme.hdr"
```

## Структура базы данных EME БД

EME БД состоит из таблиц данных, которые называются "Записи". Строки таблиц — "Строки записей", колонки — "Поля записей".

**Синонимы для поиска:** структура БД, записи EME, поля записи, таблицы EME, атрибуты полей

### Типы атрибутов полей

| Атрибут | Описание |
|---------|----------|
| Целое число | Integer (2 или 4 байта) |
| Число с плавающей точкой | Float/Double |
| Строка текста | String |
| Массив | Array |
| Деньги | Money |
| Ссылка | Reference на другую запись |
| Дата | Date |
| Время | Time |

Поле может быть постоянной или переменной длины. Для полей переменной длины создаётся дополнительное поле **VLP-указатель**. В одном поле может размещаться целый массив одинаковых переменных.

Структура БД записана в самой БД — запись "Записи БД" содержит названия, номера и свойства всех записей, а запись "Поля БД" — всех полей.

### Обязательные записи

Помимо системных, в EME БД есть обязательные записи, без которых программа не запустится:

- **Настройки модулей** — глобальная запись, изменение влияет на все рабочие станции и сервер
- **Настройки рабочих станций** — локальная запись, управляет работой одного компьютера
- **Классы ЕМЕли** — хранят прикладной EME-L код
- **Запросы SQL** — хранят SQL-запросы

### Системные записи

Системные записи определяют фундамент базы данных:

- Запись с номером `-1` — описание записей
- Запись с номером `-2` — описание полей
- Запись с номером `-3` — перечень атрибутов полей

### Системные поля

Помимо пользовательских полей, в EME БД присутствуют системные поля (создаются ядром БД):

- **Поле ключей** (4 байта) — содержит физический ключ строки. Уникален в масштабах всего проекта, включая все филиалы. Формируется как порядковый номер создания строки, маскируемый номером филиала в старших битах
- **Поле флагов** (4 байта) — битовые признаки строки: удалённость, новая строка, модифицированность (требует экспорта), экспортировалась хотя бы один раз и другие
- **Поле счётчиков ссылок** (4 байта) — количество ссылок на данную строку из других записей. Позволяет без перебора определить наличие ссылок. Используется для проверки возможности удаления строки

## Выравнивание EME БД с сервером

Выравнивание — процесс синхронизации данных между сервером и рабочими станциями.

**Синонимы для поиска:** выравнивание, синхронизация, обновление данных, align, подключение к серверу

### Процесс выравнивания

1. Подключение к серверу по IP-адресу и порту
2. Получение изменений с сервера
3. Применение изменений к локальной БД

```eme-l
'Выравнивание выполняется автоматически при:'
'1. Запуске рабочей станции в режиме /ws или /netvm'
'2. Нажатии кнопки "Подключиться к серверу" в диалоге заливки'

'Важно: база должна быть запущена с ключом /nodllalign'
'во избежание множества ошибок при выравнивании по модулям!'
```

### Предупреждения при выравнивании

- Если выровнять локальную БД (запущенную в моно-режиме) с сервером — **все локальные изменения будут потеряны**
- Если БД запускается в режиме сервера — выравнивание не требуется
- HTTP-сервер НЕ отвечает за обновление программы — для этого нужен сервер транзакций

### Пошаговое выравнивание

1. Перейти по меню: Утилиты БД → Заливка проекта → Выравнивание изменений
2. Нажать кнопку "Подключиться к серверу"
3. Нажать кнопку "Вернуться в моно"
4. Дождаться сообщения "Выполнена базовая выгрузка проекта"

### IP-адреса серверов

| Среда | IP-адрес | Порт |
|-------|----------|------|
| Релиз-версия | 10.1.15.20 | 9949 |
| Пре-прод | 10.1.15.20 | 9939 |
| Разработческая версия | 10.1.6.126 | 4546 |

**Важно:** к серверу пре-прода не следует подключаться без согласования с командой Базовой версии WMS.

### Настройка подключения к серверу

В настройках рабочей станции (Ctrl-Shift-W):

- **Сеть → IP-адрес сервера** — адрес сервера WMS
- **Сеть → Порт прослушивания** — порт сервера
- **HTTP-сервер → IP-адрес** — адрес HTTP-сервера
- **HTTP-сервер → Порт** — порт HTTP-сервера

## Связи между записями в EME БД

В EME БД отсутствуют индексы и внешние ключи как в классических СУБД. Связи реализуются через ссылочные поля.

**Синонимы для поиска:** связь записей, ссылочные поля, цепочки, один-ко-многим, ссылка, reference

### Типы ссылочных полей

| Тип | Описание |
|-----|----------|
| Ссылка на начало цепочки | Указывает на первую строку в цепочке |
| Ссылка на заголовок цепочки | Указывает на родительскую запись |
| Ссылка (простая) | Указывает на произвольную строку |

### Создание связи "один-ко-многим"

Для создания связи между записями (например, Клиенты → Звонки клиентов):

1. В родительской записи создать поле типа "Ссылка на начало цепочки"
2. В дочерней записи создать поле типа "Ссылка на заголовок цепочки"
3. Создать сортировку-цепочку в дочерней записи

```eme-l
'Пример структуры связи:'
'Запись "Клиенты":'
'  - Фамилия (Строка текста)'
'  - @Звонки клиентов (Ссылка на начало цепочки)'

'Запись "Звонки клиентов":'
'  - Дата (Дата)'
'  - Время (Время)'
'  - @Клиент (Ссылка на заголовок цепочки)'
'  - Сортировка по клиенту (Сортировка-цепочка)'
```

## Транзакции в EME БД

Транзакция — перевод БД из одного целостного состояния в другое целостное.

**Синонимы для поиска:** транзакция, transaction, целостность данных, атомарность, commit rollback

### Механизмы транзакций

1. **Повторно-входовой файл намерений** — запись изменений перед выполнением
2. **Предварительное дублирование** — бэкап изменяемых сегментов

### Просмотр серверных транзакций

Диалог "Просмотр серверных транзакций" доступен по пути: Система → Ресурсы → Просмотр серверных транзакций

Позволяет анализировать:
- Номер транзакции
- Дату и время
- Название транзакции
- Данные внутри транзакции
- Запись и поле БД

## Чтение и запись данных в EME БД

EME БД относится к системам класса RISC (сокращённый набор команд).

**Синонимы для поиска:** put_data, get_data, чтение данных, запись данных, RISC, прямой доступ

### Команды чтения и записи

```eme-l
'Команды put_data() и get_data() работают напрямую с адресом блока данных'
'Параметры:'
'- Номер Записи'
'- Номер Поля'
'- Физический номер строки'
'- Адрес блока данных в памяти'
'- Размер блока'
```

Для `put_data()` адрес содержит готовый блок данных для загрузки в БД. Для `get_data()` используется ссылка на свободный буфер памяти для выгрузки данных из БД. Запись и чтение выполняются очень быстро и напрямую, без операций поиска и сравнения ключей.

### Преимущества подхода

- Быстрый прямой доступ без поиска и сравнения ключей
- Минимальные накладные расходы
- Эффективная работа с большими объёмами данных

## Логирование в EME БД

Директория с логами находится внутри директории с данными (dat0ws). Основной лог-файл — CATASTRF.log.

**Синонимы для поиска:** логи, CATASTRF.log, журнал работы, логирование, debug log

### Расположение логов

| Компонент | Путь к логам |
|-----------|-------------|
| Рабочая станция | `dat0ws/logs/CATASTRF.log` |
| Сервер | `dat_server/logs/CATASTRF.log` |
| HTTP-сервер | `dat_http/logs/CATASTRF.log` |
| Мозговой компьютер | `dat_bc/logs/CATASTRF.log` |

### Анализ логов

```eme-l
'Примеры сообщений в CATASTRF.log:'
'- "Сервер транзакций загружен. Интерфейс {ip:port}"'
'- "WEB-сервер транзакционных запросов загружен"'
'- "Мозговой компьютер EME загружен"'
'- "Контроль сервисов..." (выводится каждую минуту)'
```

## Мониторинг и диагностика

Для мониторинга работы EME.WMS используются различные инструменты.

**Синонимы для поиска:** мониторинг, диагностика, консоль сервера, статистика, производительность

### Консоль сервера

Через диалог "ЕМЕ БД сервер менеджер" (service.exe):

- Вкладка "Пользователи" — подключенные пользователи и устройства
- Вкладка "Транзакции" — статистика по транзакциям
- Вкладка "Журнал работы" — вывод из CATASTRF.log
- Вкладка "Настройки" — настройки рабочей станции

### Тест производительности

Запуск теста: Система → Тест сети → Тест скорости компьютера

- Рекомендуемая производительность: 300% от базовой
- Минимальная производительность: 60% от базовой
